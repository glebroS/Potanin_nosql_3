## 1. Схема графа (Частина 1)

### ASCII-діаграма схеми графа
```text
   (User) {userId, gender, age, occupation}
      |
      | [RATED] {rating, timestamp}
      v
   (Movie) {movieId, title}
      |
      | [HAS_GENRE]
      v
   (Genre) {name}
```

### Обґрунтування рішень проєктування

#### 1. Які сутності стали вузлами, а які — ребрами? Чому?
*   **Вузли (Nodes):** `User`, `Movie`, `Genre`. Вони представляють основні фізичні та логічні сутності доменної області. Користувачі та Фільми мають власні набори характеристик та ідентифікатори. Жанри виділені в окремі вузли, оскільки вони виступають логічними категоріями, які зв'язують фільми між собою.
*   **Ребра (Edges):**
    *   `RATED`: Напрямлений зв'язок від `User` до `Movie`. Моделює дію оцінювання користувачем конкретного фільму.
    *   `HAS_GENRE`: Напрямлений зв'язок від `Movie` до `Genre`. Моделює класифікацію фільму за жанрами.

#### 2. Оцінка користувача за фільм — це ребро `(User)-[:RATED]->(Movie)` чи окремий вузол `(Rating)`?
У даному проєкті обрано варіант **ребра (relationship)**. Нижче наведено аргументацію та trade-off-и обох підходів:

*   **Підхід А: Оцінка як ребро `(User)-[:RATED {rating, timestamp}]->(Movie)` (Обраний)**
    *   *Переваги:* Схема є максимально природною та простою для читання. Швидкість траверсингу (обходу) між користувачами та фільмами є максимальною, оскільки ми переходимо напряму через одне ребро (1 хоп). Це суттєво оптимізує рекомендаційні Cypher-запити.
    *   *Недоліки:* Якщо у майбутньому знадобиться пов'язати оцінку з іншими сутностями (наприклад, платформа перегляду, геолокація, пристрій), Neo4j не дозволяє створювати ребра, що вказують на інші ребра. Також при великій кількості ребер на один вузол (супервузли) обхід може сповільнюватися.
*   **Підхід Б: Оцінка як окремий вузол `(User)-[:WROTE]->(Rating)-[:FOR_MOVIE]->(Movie)`**
    *   *Переваги:* Дозволяє створювати "гіперребра". Вузол `Rating` може мати власні зв'язки з іншими сутностями (наприклад, `(Rating)-[:MADE_ON]->(Device)`). Це корисно для складного логування та аудиту подій.
    *   *Недоліки:* Подвоює кількість хопів при будь-якому пошуку рекомендацій (замість одного кроку `User -> Movie` потрібно робити два: `User -> Rating -> Movie`). Це збільшує споживання пам'яті та значно уповільнює виконання запитів обходу графа.

#### 3. Чому жанри фільму вигідніше зберігати як окремі вузли `(Genre)`, а не як список у властивості вузла `Movie`?
*   **Індексна незалежність та швидкість пошуку:** Якщо жанри зберігаються списком у властивості фільму (наприклад, `genres: ["Action", "Sci-Fi"]`), то для знаходження фільмів певного жанру база даних має просканувати всі фільми або використовувати дорогі текстові індекси. З вузлами `Genre` ми робимо миттєвий перехід за вказівниками: `(Genre {name: "Thriller"})<-[:HAS_GENRE]-(m:Movie)`.
*   **Index-Free Adjacency (Безіндексна суміжність):** Це дозволяє робити складні рекомендаційні переходи через спільні жанри за $O(1)$ на кожен крок обходу (наприклад, знайти інші фільми того ж жанру: `(m1)-[:HAS_GENRE]->(g)<-[:HAS_GENRE]-(m2)`).
*   **Гнучкість схеми:** До окремого вузла `Genre` легко додавати нові зв'язки (наприклад, зв'язати жанри між собою за схожістю або створити зв'язок `(User)-[:PREFERS]->(Genre)` для профілювання інтересів).

## 2. Завантаження даних (Частина 2)

Конвертація даних виконується за допомогою скрипта [convert.py](file:///c:/Users/User/Desktop/NoSQL%203/convert.py) відповідно до вимог:
*   `movies.csv` містить поля `movieId`, `title`, `genres`.
*   `users.csv` містить поля `userId`, `gender`, `age`, `occupation` (без zip-коду).
*   `ratings.csv` містить поля `userId`, `movieId`, `rating`, `timestamp`.

Скрипт та інструкції завантаження збережені у файлі [queries/part2_load.cypher](file:///c:/Users/User/Desktop/NoSQL%203/queries/part2_load.cypher).

### Пояснення запитів завантаження:

1.  **Створення обмежень унікальності (Constraints):**
    ```cypher
    CREATE CONSTRAINT IF NOT EXISTS FOR (u:User) REQUIRE u.userId IS UNIQUE;
    CREATE CONSTRAINT IF NOT EXISTS FOR (m:Movie) REQUIRE m.movieId IS UNIQUE;
    CREATE CONSTRAINT IF NOT EXISTS FOR (g:Genre) REQUIRE g.name IS UNIQUE;
    ```
    *Пояснення:* Обмеження гарантують унікальність ідентифікаторів користувачів, фільмів та назв жанрів. Крім того, Neo4j автоматично створює під ними унікальні B-Tree індекси. Це критично важливо, оскільки при створенні зв'язків `MATCH` вузлів виконується миттєво за $O(1)$ замість повного сканування бази даних ($O(N)$).

2.  **Завантаження фільмів:**
    ```cypher
    LOAD CSV WITH HEADERS FROM 'file:///movies.csv' AS row
    MERGE (m:Movie {movieId: toInteger(row.movieId)})
    SET m.title = row.title;
    ```
    *Пояснення:* Читає CSV фільмів, перетворює `movieId` на ціле число і використовує `MERGE` для створення вузла. Якщо вузол вже існує, він просто оновлює властивість `title`, запобігаючи дублюванню.

3.  **Створення жанрів та зв'язків з фільмами:**
    ```cypher
    LOAD CSV WITH HEADERS FROM 'file:///movies.csv' AS row
    WITH row, split(row.genres, '|') AS genreList
    UNWIND genreList AS genreName
    MERGE (g:Genre {name: genreName})
    WITH row, g
    MATCH (m:Movie {movieId: toInteger(row.movieId)})
    MERGE (m)-[:HAS_GENRE]->(g);
    ```
    *Пояснення:* Функція `split` ділить рядок жанрів (наприклад, "Action|Comedy") на масив. `UNWIND` розгортає масив у окремі рядки. `MERGE` створює унікальний вузол `Genre`, а потім пов'язує його з фільмом зв'язком `HAS_GENRE`.

4.  **Завантаження користувачів:**
    ```cypher
    LOAD CSV WITH HEADERS FROM 'file:///users.csv' AS row
    MERGE (u:User {userId: toInteger(row.userId)})
    SET u.gender = row.gender,
        u.age = toInteger(row.age),
        u.occupation = toInteger(row.occupation);
    ```
    *Пояснення:* Створює вузли користувачів, перетворюючи числові значення на інтеджери. Zip-код ігнорується згідно з інструкцією.

5.  **Завантаження оцінок через `apoc.periodic.iterate`:**
    ```cypher
    CALL apoc.periodic.iterate(
      "LOAD CSV WITH HEADERS FROM 'file:///ratings.csv' AS row RETURN row",
      "MATCH (u:User {userId: toInteger(row.userId)})
       MATCH (m:Movie {movieId: toInteger(row.movieId)})
       MERGE (u)-[r:RATED]->(m)
       SET r.rating = toInteger(row.rating),
           r.timestamp = toInteger(row.timestamp)",
      {batchSize: 10000, parallel: false}
    );
    ```
    *Пояснення:* Оскільки файл оцінок великий (1 млн записів), завантаження однією транзакцією призведе до переповнення оперативної пам'яті. APOC розбиває завантаження на батчі по 10,000 рядків. 
    *Чому `parallel: false`?* При паралельному виконанні кілька потоків можуть одночасно намагатися заблокувати або оновити одні й ті самі вузли `User` чи `Movie` для створення зв'язків. Це призведе до виникнення взаємних блокувань (deadlocks) або помилок очікування локів (lock acquisition timeouts). Параметр `parallel: false` гарантує послідовне та безпечне виконання батчів в одному потоці.

## 3. Cypher-запити (Частина 3)

Всі запити збережені у файлі [queries/part3.cypher](file:///c:/Users/User/Desktop/NoSQL%203/queries/part3.cypher). Нижче наведено коди, логіку та результати їхнього виконання на базі даних AuraDB.

### Запит 1. Знайти всі фільми жанру «Thriller» із середнім рейтингом вище 4.0
*   **Cypher:**
    ```cypher
    MATCH (m:Movie)-[:HAS_GENRE]->(g:Genre {name: "Thriller"})
    MATCH (u:User)-[r:RATED]->(m)
    WITH m, avg(r.rating) AS avgRating, count(r) AS ratingCount
    WHERE avgRating > 4.0
    RETURN m.movieId AS id, m.title AS title, round(avgRating, 2) AS avgRating, ratingCount
    ORDER BY avgRating DESC;
    ```
*   **Логіка:** Спочатку відфільтровуються фільми, які належать до жанру "Thriller". Потім для кожного такого фільму збираються всі оцінки користувачів, обчислюється їхнє середнє значення. Результати фільтруються за середньою оцінкою > 4.0 та сортуються за спаданням рейтингу.

### Запит 2. Знайти користувачів, які поставили оцінку 5 більш ніж 50 фільмам

*   **Логіка:** Запит шукає ребра `RATED` з властивістю `rating = 5`, групує за користувачами та підраховує кількість таких оцінок. Фільтрує тих, у кого кількість перевищує 50.

### Запит 3. Знайти фільми, які обидва користувачі (userId=1 і userId=2) оцінили високо (рейтинг ≥ 4)

*   **Логіка:** Запит виконує перетин зв'язків від двох конкретних користувачів до спільного фільму `m`, накладаючи фільтр на значення оцінки для обох ребер.

### Запит 4. Знайти жанри, чиї фільми стабільно отримують високі оцінки — середній рейтинг і кількість оцінок

*   **Логіка:** Запит проходить від жанру до фільмів і далі до оцінок користувачів. Агрегує загальну кількість оцінок та обчислює середній бал для кожного жанру.

### Запит 5. Рекомендація «користувачі зі схожими смаками також дивилися»
*   **Cypher:**
    ```cypher
    MATCH (u1:User {userId: 1})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(u2:User)
    WHERE r1.rating >= 4 AND r2.rating >= 4 AND u1 <> u2
    WITH u1, u2, count(m) AS similarity
    ORDER BY similarity DESC
    LIMIT 10
    MATCH (u2)-[r3:RATED]->(m2:Movie)
    WHERE r3.rating >= 4 AND NOT (u1)-[:RATED]->(m2)
    RETURN m2.movieId AS id, m2.title AS title, count(u2) AS recommendationScore, round(avg(r3.rating), 2) AS avgRating
    ORDER BY recommendationScore DESC, avgRating DESC
    LIMIT 10;
    ```
*   **Логіка:** Колаборативна фільтрація. Спочатку знаходяться топ-10 користувачів (`u2`), які мають найбільше перетинів за високо оціненими фільмами з `User 1` (показник `similarity`). Далі вибираються фільми `m2`, які ці схожі користувачі оцінили на `>= 4`, але які `User 1` ще не бачив. Результати сортуються за кількістю голосів схожих користувачів.
*   **Результат виконання (Топ-5):**

    | :--- | :--- | :--- | :--- |
    | 953 | It's a Wonderful Life | 10 | 4.90 |
    | 50 | Usual Suspects, The | 10 | 4.80 |
    | 1307 | When Harry Met Sally... | 10 | 4.80 |
    | 1200 | Aliens | 10 | 4.80 |
    | 1198 | Raiders of the Lost Ark | 10 | 4.70 |

### Запит 6. Найкоротший ланцюжок зв’язку між двома користувачами (User 1 та User 100) через спільні фільми
*   **Cypher:**
    ```cypher
    MATCH p = shortestPath((u1:User {userId: 1})-[:RATED*..6]-(u2:User {userId: 100}))
    RETURN p;
    ```
*   **Логіка:** Знаходить шлях мінімальної довжини у неорієнтованому графі зв'язків `RATED` між двома вершинами, обмежуючи глибину пошуку до 6 ребер.
*   **Результат виконання:**
    `User 1 -[RATED 4]-> Movie 'Star Wars: Episode IV - A New Hope' -[RATED 4]-> User 100` (довжина шляху = 2 ребра, 1 проміжний вузол фільму).

#### Інтерпретація довжини шляху:
*   **Один хоп (довжина 1):** Неможлива між двома користувачами, оскільки користувачі не мають прямих зв'язків між собою (граф дводольний `User - Movie`).
*   **Довжина шляху 2:** Користувачі А та Б безпосередньо оцінили один і той самий фільм. Шлях: `User A -> Movie X -> User B`. Це вказує на прямий перетин інтересів.
*   **Довжина шляху 4:** Користувачі А та Б не мають спільних фільмів, але пов'язані через посередника В. Шлях: `User A -> Movie X -> User C -> Movie Y -> User B`. Це означає, що А і C дивилися фільм X, а C і Б дивилися фільм Y.
*   **Довжина шляху 6:** Зв'язок через двох посередників. Шлях: `User A -> Movie X -> User C -> Movie Y -> User D -> Movie Z -> User B`. Це демонструє далеку схожість або транзитивний зв'язок у мережі рекомендацій.

### 2. Чому запит, що зачіпає супервузол, працює повільніше, ніж запит по «звичайному» вузлу?
Індекси в Neo4j допомагають лише швидко знайти початкову точку (вузол). Проте, коли починається обхід графу (траверсинг), Neo4j зчитує суміжні ребра з фізичної пам'яті. 
*   Для звичайного вузла (наприклад, фільм з 5 оцінками) база даних зчитує 5 адрес у пам'яті за $O(1)$ часу.
*   Для супервузла (наприклад, користувач з 1,521 оцінкою або жанр Drama з 1,603 фільмами) база має завантажити тисячі адрес. Якщо ми шукаємо шляхи глибиною більше 2-3 хопів, зачіпання супервузла створює **комбінаторний вибух** (алгоритм змушений перебирати тисячі гілок шляхів). Це призводить до різкого зростання CPU-навантаження та використання оперативної пам'яті (Heap RAM).

### 3. Яку конкретну стратегію мінімізації супервузлів застосувати?
Для нашого датасету найкраще підходять такі стратегії:
*   **Траверсинг із фільтрацією на рівні зв'язків (Relationship Filtering):** Замість обходу всіх ребер `[:RATED]` ми маємо відсікати нерелевантні переходи ще на етапі запиту (наприклад, переходити тільки по ребрах, де оцінка `rating >= 4`). Це зменшує кількість відгалужень у рази.
*   **Денормалізація / Дублювання зв'язків (Split Relationships):** Для жанрів (оскільки Drama і Comedy мають тисячі ребер) можна створити більш специфічні зв'язки, наприклад: замість одного загального `HAS_GENRE` використовувати `DRAMA_GENRE`, `COMEDY_GENRE`. Тоді при пошуку комедій база даних не буде перебирати ребра драм.
*   **Усунення жанрових вузлів з транзитивних пошуків шляхів:** При пошуку зв'язків між користувачами заборонити обхід через вузол `Genre`. Траверсинг повинен йти виключно за шаблоном `(User)-[:RATED]->(Movie)<-[:RATED]-(User)`.

### 5.1. PageRank на графі фільмів
*   **Суть алгоритму:** Ми зв'язали фільми ребрами `CO_RATED` (якщо принаймні один користувач оцінив обидва фільми на `>= 4`). Вага ребра (`weight`) — кількість таких спільних користувачів.
*   **Інтерпретація високого PageRank:** 
    Високий бал PageRank у цьому графі означає, що фільм є не просто "популярним за кількістю голосів", а виступає **центральним мостом (хабом)** у мережі вподобань. Це фільм, який часто оцінюють високо разом з багатьма іншими хорошими фільмами різних жанрів. Він об'єднує навколо себе різні користувацькі кластери (наприклад, *Star Wars: Episode V* або *American Beauty*), що робить його ключовою точкою входу для будь-якої холодної рекомендації.

### 5.2. Виявлення спільнот (Louvain)
*   **Суть алгоритму:** Групує користувачів у кластери за щільністю зв'язків `SIMILAR` (спільні вподобання).
*   **Аналіз кластерів за жанрами:**
    На нашому датасеті Louvain виділяє чіткі когорти інтересів:
    1.  **Кластер 1 (Sci-Fi / Action / Adventure):** Користувачі, чиї топ-жанри за високими оцінками — Action, Sci-Fi та Adventure (шанувальники Star Wars, Terminator, Matrix).
    2.  **Кластер 2 (Drama / Romance):** Користувачі, які віддають перевагу драмам, мелодрамам та класиці (American Beauty, To Kill a Mockingbird).
    3.  **Кластер 3 (Comedy / Children's / Animation):** Користувачі, які дивляться сімейні фільми та комедії (Toy Story, Babe).

*   **Як це було перевірено?**
    Ми призначили кожному користувачу властивість `communityId` за допомогою процедури `gds.louvain.write`. Потім виконали агрегацію, підрахувавши частоту зв'язків від користувачів кожної спільноти до жанрів фільмів, які вони оцінили на `>= 4` (Query 4 в секції Louvain). Розподіл частот підтвердив виражені домінантні жанри в кожному кластері.

### 5.3. Найкоротший шлях (Dijkstra)
*   **Суть алгоритму:** Знаходить зважений шлях між двома користувачами на графі схожості.
*   **Властивість `cost`:** Оскільки вага `weight` нашого ребра — це схожість (чим більше, тим краще), для алгоритму Дейкстри (який шукає мінімум вартості) ми перетворили її на вартість: `cost = 1000.0 / weight`. Відповідно, чим більше фільмів користувачі оцінили спільно, тим менша "вартість" ребра між ними, і тим швидше Dijkstra прокладе через них шлях.
*   **Тісний світ (Small World):**
    Датасет демонструє феномен "тісного світу". Середня довжина шляху (кількість переходів) між будь-якими двома випадковими користувачами через схожість смаків становить **від 2 до 4 кроків**. Гіпотеза "шести рукостискань" повністю підтверджується із великим запасом — практично будь-які два профілі смаків можна зв'язати через максимум 3 проміжних користувачів, які мають спільні високі оцінки фільмів.

## 6. Аналіз і висновки (Частина 6)

### 1. Граф vs SQL
У частині 3 запити **Query 5 (Колаборативна фільтрація)** та **Query 6 (Пошук найкоротшого шляху)** було б вкрай складно реалізувати в SQL, а у випадку необмеженого пошуку шляху — практично неможливо з точки зору продуктивності.

*Приклад еквівалента Query 5 (Колаборативна рекомендація) в SQL:*
Щоб знайти фільми для користувача 1 через схожих користувачів, нам довелося б написати запит із 4-ма JOIN-ами таблиці `ratings` самої на себе:
```sql
SELECT r3.movie_id, COUNT(DISTINCT r3.user_id) AS score, AVG(r3.rating) AS avg_rating
FROM ratings r1
JOIN ratings r2 ON r1.movie_id = r2.movie_id AND r1.user_id <> r2.user_id
JOIN ratings r3 ON r2.user_id = r3.user_id
WHERE r1.user_id = 1 AND r1.rating >= 4 AND r2.rating >= 4 AND r3.rating >= 4
  AND r3.movie_id NOT IN (SELECT movie_id FROM ratings WHERE user_id = 1)
GROUP BY r3.movie_id
ORDER BY score DESC, avg_rating DESC
LIMIT 10;
```
*Чому це проблема для SQL:*
Цей запит вимагає сканування та з'єднання таблиці `ratings` (яка містить 1 мільйон рядків) самої з собою кілька разів. Навіть з індексами СКБД буде змушена створювати гігантські тимчасові таблиці в пам'яті (або на диску), виконувати сортування (Sort) та групування (HashAggregate). Час виконання такого запиту на реальних обсягах вимірюється секундами або хвилинами.
У Neo4j Cypher цей запит проходить безпосередньо по фізичних адресах пам'яті суміжних ребер за лічені мілісекунди. Що стосується пошуку найкоротшого шляху довільної довжини (Query 6), в SQL це вимагало б написання складної рекурсивної конструкції `WITH RECURSIVE (CTE)`, яка б "поклала" реляційну базу даних на великій глибині обходу.

### 2. Де граф програє?
Реляційна модель набагато краще підходить для наступних задач:
1.  **Агрегаційні звіти та OLAP-аналітика:** Наприклад, обчислити щомісячну сумарну кількість оцінок, розподіл оцінок за професіями для всіх користувачів, або експортувати весь список користувачів з фільтрацією за віком. SQL-бази даних (наприклад, PostgreSQL або ClickHouse) зберігають дані компактними рядками/колонками і вміють розпаралелювати лінійні сканування таблиць на всі ядра процесора, виконуючи такі агрегації на порядки швидше за графові бази.
2.  **Масове оновлення або видалення даних:** Запити на кшталт "видалити всіх користувачів певного віку" або "збільшити всі оцінки на 1" в SQL виконуються миттєво через лінійну модифікацію сторінок даних. В Neo4j це вимагає перебудови покажчиків зв'язків у графі, що працює повільніше і споживає багато пам'яті на транзакції.
3.  **Експорт великих масивів даних:** Для простого читання та передачі сирих табличних даних реляційні БД є стандартом де-факто завдяки простоті потокового читання.

### 3. Покращення схеми
Для прискорення запитів з частини 3 пропонуються такі зміни:
1.  **Прискорення Query 1 (Фільтрація трилерів з високим рейтингом):**
    Ми можемо денормалізувати схему, додавши властивість `genres` як масив рядків безпосередньо у вузол `Movie` (на додачу до окремого вузла `Genre`), та створити індекс `CREATE INDEX ON :Movie(genres)`. Це дозволить Query 1 миттєво знаходити фільми потрібного жанру без необхідності робити JOIN-перехід `(m)-[:HAS_GENRE]->(g)`, а відразу переходити до підрахунку рейтингів.
2.  **Прискорення Query 5 (Колаборативна фільтрація):**
    Найбільш ресурсомісткою частиною колаборативної фільтрації є динамічне обчислення перетину оцінених фільмів для пошуку схожості користувачів. Ми можемо створити прямі зв'язки схожості `(User)-[:SIMILAR {score}]->(User)`. Ці зв'язки можна вираховувати у фоновому режимі (наприклад, раз на добу за допомогою Louvain або косинусної схожості) і записувати в базу. Тоді Query 5 виконуватиметься в один крок: ми просто зчитуємо вже готові зв'язки `SIMILAR` від заданого користувача і відразу переходимо до його фільмів. Це прискорить рекомендації в сотні разів.
